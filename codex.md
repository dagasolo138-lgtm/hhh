# 中国黄金模拟市场 · Codex

> 本文件是项目长期交接中枢。开发时按需要查阅本文件和相关源文件；**每次开发完成后，必须在本文末尾追加一条版本更新概述。**

---

## 1. 项目定位

这是一个部署在 GitHub Pages 的纯前端、单标的中国黄金模拟市场。

唯一资产：**人民币计价的中国黄金现货ETF模拟份额**。

当前锁定：**上交所黄金ETF通用制度框架**。项目没有绑定某一只现实基金代码；未来选择真实参考产品后，再校准每份含金量、管理费、申赎单位、规模和历史参数。

```text
国际现货黄金（美元/盎司）
× 美元兑人民币
÷ 31.1034768
→ 人民币换算金价（人民币/克）
→ 国内黄金溢价
→ 国内黄金现货价格
→ 基金理论净值
→ 交易者订单、订单簿与撮合
→ 二级市场交易价格
→ 情绪、风险和下一刻订单
```

项目用于市场机制模拟、教育和实验，不连接真实交易，不构成投资建议。

---

## 2. 不可破坏的规则

1. 理论净值和二级市场交易价格必须分离。
2. 技术指标只能影响交易者订单，不能直接改价格。
3. 本刻价格、成交量、失衡、回撤和波动率只能在下一刻反馈，禁止同刻自循环。
4. 同一初始状态、随机种子和调用顺序必须复现同一世界线。
5. 自动交易者、做市商和玩家之间的现金与份额必须守恒。
6. 乐观度、注意力、分歧度、脆弱度、风险偏好和信息不对称属于隐藏状态，页面只能展示公开数据和推断摘要。
7. 智能分析只能读取市场资料，不能改价格、订单簿、账户、随机种子或存档。
8. 市场推进、公开事件、玩家成交、玩家委托、撤单和订单过期都必须写日志。
9. 二级市场交易、现金申赎、黄金现货合约申赎必须是三套独立流程。
10. 静态网页没有服务器数据库；重要世界线必须导出 JSON 存档。

## 3. 交易制度基准

### 制度文件

- `交易制度参数表.md`：人读制度表与实现缺口。
- `市场/交易制度参数.js`：机器可读取的制度参数唯一来源。
- `市场/制度执行器.js`：状态、订单校验、价格边界、冻结资产。
- `市场/制度收尾.js`：推进后刷新日线边界、限制价格、规范盘口、累计当日成交。
- `市场/制度一致性检查.js`：制度边界、冻结资产、订单状态与数量守恒检查。

### 当前已执行规则

| 规则 | 当前处理 |
|---|---|
| 最小报价单位 | 0.001 元；交易价、盘口和限价单均校验。 |
| 买入数量 | 最低 100 份，必须是 100 份整数倍。 |
| 卖出数量 | 支持零股卖出。 |
| 当日回转 | 玩家买入后可当日卖出。 |
| 涨跌幅 | 以前收盘价为基准，±10%；交易价、盘口和限价价格都受边界限制。 |
| 连续竞价 | 09:30–11:30、13:00–15:00；即时市价成交和限价单提交只在此时段开放。 |
| 集合竞价 | 09:15–09:25；当前禁止即时成交，独立订单池尚未实现。 |
| 限价有效期 | 当日有效；跨交易日自动失效并释放冻结资产。 |
| 大宗交易 | 当前不模拟。 |

### 未实现制度

- 独立集合竞价订单池、单一开盘价算法。
- 真实节假日、停复牌、异常波动和特殊首日规则。
- 现金申赎、黄金现货合约申赎、申赎清单、额度、清算与交收。
- 手续费、申赎费、税费、融资和保证金。

  ├─ 制度收尾
  ├─ 跨日委托失效
  ├─ 玩家限价队列撮合
  ├─ 写入上一刻观测
  └─ 制度一致性检查
  ↓
市场/演算器.js
  ↓

### 页面层

- `index.html`：中文页面结构。
- `样式.css`：主视觉和移动端基础布局。
- `委托样式.css`：限价委托、冻结资产、待处理订单和撤单区域。
- `界面.js`：按钮绑定、渲染、存档、导入导出、限价单与撤单操作。
- `核心.js`：页面唯一业务出口。页面不得越过它直接修改市场状态。

### 市场层

- `市场/状态工厂.js`：创建版本 2 市场状态、交易时段、资产基础定义。
- `市场/演算器.js`：全球环境、国内映射、心理状态、自动交易者、订单簿、自动撮合、统计、历史。
- `市场/时序演算器.js`：市场推进总入口。
- `市场/玩家交易.js`：玩家市价单。
- `市场/玩家限价委托.js`：玩家限价单、撤单、冻结、价格时间优先、跨日失效。
- `市场/制度执行器.js`：制度执行。
- `市场/制度收尾.js`：价格边界和盘口规范。
- `市场/一致性检查.js`：公式、盘口、现金份额守恒、时序检查。
- `市场/制度一致性检查.js`：涨跌停、最小报价单位、冻结资产与委托守恒检查。
- `市场/自检.js`：无界面回归自检。

## 6. 限价委托与撮合逻辑

入口：`市场/玩家限价委托.js`。

### 提交

1. 校验连续竞价、方向、买入数量、限价最小变动和涨跌停范围。
2. 买单检查可用现金，卖单检查可用持仓。
3. 冻结资产。
4. 创建当日有效限价单并写入待处理队列。
5. 写入“提交委托”日志。
6. 立刻尝试与当前对手盘口成交。

### 排序规则

- 买单：价格从高到低，同价按 `委托序号` 从小到大。
- 卖单：价格从低到高，同价按 `委托序号` 从小到大。
- 同价订单先到先成交。

### 成交规则

- 买单只吃 `卖方价格 ≤ 买入限价` 的档位。
- 卖单只吃 `买方价格 ≥ 卖出限价` 的档位。
- 不可成交部分继续保留在队列。
- 每个成交都会更新玩家、做市商、成交量、成交额、订单失衡、当日统计、交易价和日志。

### 撤单与过期

- 撤单：只能撤销待处理订单，立即释放剩余冻结资产。
- 过期：时序推进检测到交易日变化时，将旧交易日待处理订单标记为已过期并释放冻结资产。

### 当前边界

当前只有一个玩家账户，因此不会出现不同玩家之间的交叉限价单撮合；玩家限价单主要与自动做市盘口成交。多账户撮合是后续扩展，不应在当前版本假装已实现。

## 7. 一刻市场推进顺序

1. 初始化制度状态、守恒基线、上一刻观测。
2. 记录推进前的中国交易日。
3. 自动市场推进：时间、事件、全球黄金、汇率、国内金价、净值、心理、自动订单、自动盘口、自动撮合、统计。
4. 制度收尾：跨日更新前收盘和涨跌停，限制交易价，规范盘口，累计当日成交。
5. 使跨日玩家委托失效。
6. 在当前连续竞价盘口中撮合玩家限价队列。
7. 写入上一刻观测。
8. 执行市场一致性与制度一致性检查。

时序权威：`市场/时序因果图.js`。

## 10. 测试与验证

本地命令：

```text
npm run 自检
```

当前自检覆盖：

- 相同种子复现；不同种子产生不同世界线。
- 集合竞价拒绝即时市价成交。
- 非 100 份整数倍买入被拒绝。
- 市价买入和卖出后的现金与份额守恒。
- 交易价和盘口报价不突破涨跌停边界。
- 两笔同价限价买单排队时，先提交的订单先成交。
- 部分成交后冻结现金正确保留。
- 撤单后冻结资产归零。
- 市场一致性与制度一致性检查通过。

GitHub Actions：`.github/workflows/市场自检.yml` 已配置，但本对话尚未获得可靠云端执行结果。不得写成“云端已通过”。
