## 1. 项目定义

这是一个部署在 GitHub Pages 的纯前端、单标的中国黄金模拟市场。

唯一可交易资产：**人民币计价的中国黄金现货ETF模拟份额**。

当前锁定的制度框架：**上海证券交易所黄金ETF通用制度**。项目暂不绑定某一只具体基金代码；未来选择真实参考产品后，再单独校准每份含金量、管理费、申赎单位、基金规模与初始数据。

市场主链：

```text
国际现货黄金（美元/盎司）
× 美元兑人民币
÷ 31.1034768
→ 人民币换算金价（人民币/克）
→ 国内黄金溢价
→ 国内黄金现货价格
→ 基金理论净值
→ 交易者订单与订单簿撮合
→ 二级市场交易价格
→ 成交、情绪、风险和下一刻订单行为
```

目标是构建可复盘的市场机制，不是随机涨跌演示器，也不连接真实交易。

## 2. 不可破坏的世界规则

1. **理论净值与交易价格必须分离。**
   - 理论净值来自境内黄金现货、每份黄金含量、跟踪误差与费用拖累。
   - 二级市场交易价格来自订单簿、成交、流动性和溢价折价。

2. **技术指标不能直接改价格。**
   均线、动量、成交量和波动带只能影响交易者的订单与目标仓位。

3. **本刻结果只能在下一刻反馈。**
   收益、成交量、订单失衡、回撤和波动率不能回头影响本刻已经形成的订单。

4. **同一种子必须可复现。**
   相同初始状态、随机种子和调用顺序应得到同一条市场世界线。

5. **现金与基金份额必须守恒。**
   自动交易者、做市商和玩家之间每一笔资金与份额转移必须有对手方。

6. **玩家不能直接看到隐藏状态。**
   乐观度、注意力、分歧度、脆弱度、风险偏好和信息不对称属于后台状态；页面只展示公开数据和推断摘要。

7. **智能分析只能读取。**
   智能分析不能改价格、随机种子、订单簿、账户、存档或交易制度。

8. **重要变化必须有日志。**
   市场推进、公开事件、玩家成交、订单状态和制度拒绝原因都应可回放。

9. **二级市场交易、现金申赎、黄金现货合约申赎必须分开。**
   它们的资格、时段、交收和可用份额规则不同，禁止用同一个“买卖”流程混合实现。

10. **静态网页没有服务器数据库。**
    本地存档只能在同浏览器环境中通常恢复；重要世界线必须导出。

## 3. 交易制度基准

权威文件：

- `交易制度参数表.md`：面向人阅读的制度表、已实现状态与待实现缺口。
- `市场/交易制度参数.js`：机器可读取的制度参数唯一来源。

当前锁定的上交所黄金ETF二级市场规则：

| 规则 | 参数 | 当前实现状态 |
|---|---:|---|
| 当日回转 | 当日买入可当日卖出 | 玩家市价交易已符合 |
| 最小报价单位 | 0.001元 | 当前资产定义已使用；待统一读取制度参数 |
| 买入最小数量 | 100份 | 当前已使用 |
| 买入数量 | 100份整数倍 | 当前已使用 |
| 零股卖出 | 允许 | 待限价单层实现 |
| 涨跌幅 | 相对前收盘价 ±10% | 参数已锁定，尚未接入引擎 |
| 集合竞价 | 09:15–09:25 | 当前仅有阶段名，未独立撮合 |
| 连续竞价 | 09:30–11:30、13:00–15:00 | 时钟已覆盖，撮合待升级 |
| 大宗交易 | 15:00–15:30 | 当前不模拟 |

黄金ETF申赎规则：

- 现金申赎：09:30–11:30、13:00–15:00；属于独立业务，不作为普通玩家交易按钮。
- 黄金现货合约申赎：09:30–11:30、13:00–15:00；需要证券账户、黄金账户、备案和相应会员通道。
- 普通模拟玩家暂不开放两类申赎；未来只作为授权参与者或套利资金的后台机制。

### 当前制度缺口

当前页面不能宣称“完全复刻真实上交所黄金ETF”，因为仍缺：

- 上一交易日收盘价、涨停价和跌停价状态。
- 撮合前的 ±10% 价格边界校验。
- 独立集合竞价订单池和开盘单一价格算法。
- 限价单、撤单、价格优先、时间优先队列。
- 日历、停复牌、异常状态、申赎清单、额度和结算。
- 具体基金合同参数和真实公开数据校准。

## 7. 关键状态与变量

顶层状态：

```text
状态 = {
  版本: 2,
  资产,
  系统,
  变量,
  派生,
  交易者,
  订单簿,
  事件,
  结构,
  历史
}
```

### `资产`

当前默认：

- 名称：`中国黄金现货指数基金`
- 简称：`中国黄金`
- 计价货币：人民币
- 最小交易单位：100份
- 价格最小变动：0.001元
- 每份黄金含量：0.01克/份（模拟默认值，非具体基金合同参数）

### `系统`

- `随机种子`：决定世界线可复现性。
- `演算序号`：每推进一刻加一。
- `虚拟时间`：模拟时钟。
- `交易日序号`：低频事件和季节性基准。
- `盘中阶段`：集合竞价、上午交易、午间休市、下午交易、收盘后、休市。
- `国际市场阶段`：亚洲、伦敦、纽约、重叠、国际休市。
- `是否可交易`：玩家订单先检查。
- `守恒基线`：自动交易者与玩家合计现金、份额的初始总额。

### `变量`

变量必须先登记在 `变量字典.js`。主要分组：

- 全球黄金：国际现货黄金、实际利率冲击、美元强弱、通胀预期、地缘风险、全球资金流、条件方差。
- 汇率与中国映射：美元兑人民币、人民币换算金价、国内黄金溢价、国内黄金现货、期现基差。
- 基金：每份黄金含量、理论净值、交易价格、溢价折价、跟踪误差、申赎摩擦、套利压力。
- 情绪与风险：乐观度、注意力、分歧度、脆弱度、风险偏好、信息不对称、情绪不确定度。
- 微观结构：买卖深度、价差、撤单率、补单率、主动买卖、订单失衡、价格冲击系数、实现波动率。
- 玩家账户：玩家现金、持仓、成本、市值、浮动盈亏、总资产。

### `交易者`

| 标识 | 作用 |
|---|---|
| `价值套利` | 根据交易价与理论净值偏离形成均值回归订单。 |
| `趋势资金` | 根据技术指标、风险偏好与波动率形成趋势订单。 |
| `散户群体` | 根据乐观度、注意力、叙事与回撤形成净需求。 |
| `风险控制资金` | 根据回撤、脆弱度和波动率减仓。 |
| `信息优势资金` | 当前名称待迁移；实际含义是带噪声的先行信号与信息扩散。 |
| `做市商` | 提供双边盘口，承接成交并积累库存。 |

### `订单簿`

```text
订单簿 = {
  买方: [{ 档位, 价格, 数量 }, ...10档],
  卖方: [{ 档位, 价格, 数量 }, ...10档],
  中间价,
  上次成交价,
  版本
}
```

当前自动交易者和玩家市价单都可吃单。限价委托和挂单队列尚未实现。

## 8. 一刻市场如何推进

执行入口：`市场/时序演算器.js` 的 `推进时序中国黄金市场`。

规则：先读取上一刻观测，运行本刻市场，再写入下一刻观测。

当前顺序：

1. 推进虚拟时间，判定中国交易阶段和国际市场阶段。
2. 更新事件系统：新事件、衰减、公开延迟、泄露强度、叙事与热度。
3. 更新全球黄金与汇率。
4. 更新中国本地黄金、理论净值、期现基差与套利压力。
5. 更新隐藏心理状态。
6. 形成交易者目标仓位与订单。
7. 构建十档订单簿，计算价差、深度、撤单率、补单率与冲击系数。
8. 撮合自动交易者订单，形成主动买卖、订单失衡、成交量、成交额与做市库存。
9. 更新交易价格、收益、波动率、回撤、溢价折价与技术指标。
10. 更新成本分布、日志、快照与跨期反馈。
11. 写入下一刻观测，并执行一致性检查。

概念上可称八阶段；代码为了可维护性拆成更多函数，不能改变时序。

## 9. 同刻因果与跨期反馈

同刻因果：

```text
时间制度
→ 全球环境
→ 国内映射
→ 事件与心理
→ 交易者订单
→ 订单簿撮合
→ 市场统计
→ 存档与反馈
```

必须读取上一刻观测的反馈：

- 价格收益 → 乐观度。
- 成交量、异常成交量 → 注意力。
- 双向主动成交、订单失衡反复、波动率 → 分歧度。
- 订单簿脆弱度、回撤 → 脆弱度。
- 持仓成本分布 → 最近盈利体验 → 风险偏好。
- 回撤、波动率 → 风险控制减仓压力。
- 技术指标集合 → 下一刻趋势资金目标仓位。
- 成交结果、订单失衡 → 下一刻做市库存与报价意愿。
- 中国休市期间的国际黄金、汇率、本地溢价变化 → 隔夜信息积累 → 下一开盘跳空压力。

运行时序权威文件是 `市场/时序因果图.js`。

## 10. 存档与兼容性

- 主存储：浏览器 IndexedDB。
- 备用：浏览器 localStorage。
- 同设备、同浏览器正常刷新或重开：通常可恢复。
- 跨设备、清除站点数据、无痕模式：可能丢失。
- 导出JSON：重要世界线的可靠迁移方式。

当前中国黄金市场状态版本：`2`。

旧版华辰工业状态不能直接导入版本2市场。

已知命名债务：`存档.js` 内的数据库名称和备用键仍含“华辰工业”。功能可用，但迁移时必须保护旧世界线并处理旧键读取。

## 11. 验证与测试

本地自检：

```text
npm run 自检
```

自检至少应验证：

- 同种子可复现。
- 不同种子不会完全相同。
- 连续推进后交易价格、理论净值和盘口有效。
- 买一价低于卖一价。
- 玩家可以买入再卖出。
- 玩家成交后现金与份额守恒。
- 一致性检查通过。

GitHub Actions 工作流：`.github/workflows/市场自检.yml`。

当前已配置工作流，但本对话尚未拿到可靠云端执行结果。不得把“已配置”写成“已通过”。

每次改市场逻辑后的最低检查清单：

1. 新变量是否先登记到 `变量字典.js`。
2. 新时序关系是否先登记到 `市场/时序因果图.js`。
3. 新交易规则是否先登记到 `市场/交易制度参数.js`。
4. 是否出现同一刻自我循环。
5. 是否破坏理论净值公式。
6. 是否破坏现金或份额守恒。
7. 是否新增无法解释的直接价格修改。
8. 是否能写入完整日志。
9. 是否在本文件末尾追加版本记录。

## 12. 智能分析与安全边界

当前 `智能分析.js` 只构建资料和请求边界，不直接连接外部服务。

未来接入 DeepSeek 或其他模型时：

- 页面端不得硬编码密钥。
- 需要安全服务端或受控代理。
- 模型只能读取公开资料、允许的推断摘要和波动日志。
- 模型输出必须区分事实、推断与无法判断。
- 模型不得修改市场状态、随机种子、订单簿、玩家账户或存档。
- 模型不得输出针对现实资产的个性化买卖指令。
